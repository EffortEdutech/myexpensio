## 0. Correcting the premise first

Bismillah — before proposing anything, the starting assumption needs a correction, confirmed by reading the actual code (not docs, which don't cover this):

**`scan_service` (the Python/FastAPI app deployed on Render, `scan_service/render.yaml`, service name `myexpensio-scan`) does NOT generate the export report.** It exposes exactly three endpoints:
- `POST /process` — receipt image OCR/pre-processing (opencv)
- `POST /parse-tng` — parses a TNG e-wallet statement PDF into transactions (pdfplumber/pymupdf)
- `POST /highlight-tng` — highlights matched transaction rows on a TNG statement PDF (pymupdf annotations)

**The actual PDF/CSV/XLSX claim export is built entirely in Next.js**, in `apps/user`:
- `app/api/exports/route.ts` — the export endpoint
- `lib/pdf-builder.ts` — draws the PDF by hand using `pdfkit` (manual x/y coordinates, no templating)
- `lib/export-builder.ts` + `lib/export-columns.ts` — CSV/XLSX generation via `exceljs`
- `lib/pdf-lib`-based merge step appends the (optionally Python-highlighted) TNG bank statement PDF pages onto the end of the report

Python is only ever called for one narrow sub-step: highlighting matched rows on a *bank statement* PDF before it gets appended to the report. If that call fails for any reason, the code falls back to the un-highlighted statement — export never breaks because of Python being down. `SCAN_API_URL` / `SCAN_API_SECRET` are the env vars that wire this up (`apps/user/lib/tng-highlighter.ts`).

So: Python-on-Render is real, but it isn't "the export system" — it's a receipt-OCR + statement-parsing microservice that happens to also touch one PDF along the way. Worth knowing before deciding where AI should plug in.

## 1. The JSON format already prepared

Two related shapes make up the existing contract — this is what any AI-assisted step would need to consume or produce, unchanged:

**Mobile → web wire payload** (`apps/user-mobile-v2/src/features/exports/types.ts`), `ExportPreviewPayload`:
```
{
  claims: ExportPreviewClaim[],
  rows: ExportPreviewRow[],
  generatedAt: string
}
```
Sent as `mobile_payload` in the POST body to `/api/exports`, alongside `mobile_tng_statements: { statementLabel, pdfBase64, items: [{ transNo, amountCents }] }[]`.

**Server-side render shape** (`apps/user/lib/pdf-builder.ts`), built from the above via `buildPdfDataFromMobilePayload()`:
```
PdfData = {
  org_name, claimer_name, claimer_email, claimer_department,
  claimer_location, claimer_company_name, generated_at,
  layout: PdfLayoutConfig,
  claims: PdfClaim[],
  tng_statements
}

PdfClaim = { id, title, period, status, submitted_at, total_myr, items: PdfItem[] }

PdfItem = {
  id, type, claim_title, date, date_iso, description,
  qty, rate, amount_myr, receipt_path, tng_trans_no
}
```

This `PdfData` shape is the natural "already prepared JSON" the user means — it's deterministic, fully computed server-side before any rendering happens, and matches the locked baseline rule that exports must use submitted-snapshot values with no recalculation.

## 2. Where AI could plug in — options, not a decision

The locked baseline (`CLAUDE.md`) is explicit: *"Exports match deterministic rules and are audit-friendly"* and claim totals must never be recalculated post-submit. That constraint rules out letting an LLM compute or rewrite any number, date, or amount in the report. Any AI involvement has to sit strictly on the presentation side, never the numeric side. With that boundary fixed, here are the realistic options:

### Option A — AI as a build-time tool only (lowest risk)
Use an LLM (Claude/Gemini) *during development* to help design a better HTML/CSS export template, replacing `pdfkit`'s manual coordinate-drawing with something maintainable. No AI call happens at export time — the JSON still renders through a fixed, deterministic template. This is "AI-assisted," not "AI-generated."
- **Pros:** Zero determinism risk, zero new runtime dependency, zero added cost, fixes the real underlying pain (pdfkit's manual layout is genuinely hard to maintain).
- **Cons:** Doesn't feel like a new "feature" — it's a refactor with better tooling, not new capability.

### Option B — AI-generated narrative additions, numbers untouched
Add an optional section (e.g., a short "Expense Summary" paragraph or spending-pattern note) generated by Gemini at export time, fed *only* the already-computed `PdfData` totals as read-only input — same free-tier Gemini key pattern already used for S1/S3/S4 receipt/odometer/voice capture. AI never touches the table of claim items or totals; it only writes prose that sits alongside them, clearly labeled as AI-generated.
- **Pros:** Genuinely new value (an actual AI feature in the export), reuses existing Gemini plumbing/BYOK pattern, cheap (one short text-generation call per export, well within free tier).
- **Cons:** Any AI text near financial figures needs careful labeling/disclaimer for audit purposes; adds a runtime dependency (Gemini call) to the export path, which currently has zero external dependencies in its critical path (Python highlighting already degrades gracefully — this would need the same fallback discipline).

### Option C — Replace pdfkit with HTML+CSS rendering (biggest structural change)
Move report rendering from `pdfkit`'s manual drawing to an HTML/CSS template rendered to PDF, which is far easier to maintain and style well. Two sub-paths:
- **C1: Headless Chromium in Node** (Puppeteer/Playwright + `@sparticuz/chromium` for serverless). Works on Vercel but the binary is large and cold-starts are slow on free tiers — real risk given the zero-budget constraint.
- **C2: Move rendering into `scan_service` using WeasyPrint** (pure-Python HTML/CSS → PDF, no headless browser needed, free, lightweight). This is the option that actually matches Eff's original instinct — Python-on-Render *would* own report rendering, just not the way currently assumed. Next.js would send the same deterministic `PdfData` JSON to a new `scan_service` endpoint (e.g. `POST /render-report`), which renders a Jinja2 HTML template through WeasyPrint and returns PDF bytes. AI's role here is again build-time only (helping design the Jinja2/CSS template), keeping runtime fully deterministic.
- **Pros:** Solves the real pain (pdfkit layout is brittle) with a genuinely better long-term architecture; C2 specifically reuses Render (already free-tier, already deployed) instead of adding a new service.
- **Cons:** Real migration work — every existing report layout needs re-implementing in HTML/CSS; touches the export path that's currently working and audit-relied-upon, so needs careful parallel-run testing before cutover.

## 3. What "using AI for export to PDF" most likely should NOT mean

Worth naming explicitly so it's ruled out rather than silently assumed: an LLM should not be asked to read the JSON and freely generate/format the PDF's numeric content per export (e.g., "have Gemini turn this JSON into a nicely laid-out report"). Free-form LLM generation of financial documents risks silent transcription errors in amounts/dates — exactly the failure mode the locked baseline's determinism rule exists to prevent. If AI touches layout, it should do so through a fixed, reviewed template (Option A/C), not through per-export freeform generation of the whole document.

---

## 4. Recommendation (for discussion, not decided)

Option A now (cheap, safe, fixes real pain in `pdf-builder.ts`), with Option C2 (Python/WeasyPrint on the existing Render service) as the real structural fix if/when the current manual-coordinate PDF layout becomes a bigger maintenance problem than it already is. Option B (AI narrative blurb) is a nice-to-have that could be added independently, later, once there's appetite for an AI-touches-the-report-at-all feature — it's the only option here with a genuine new user-facing AI capability, everything else is "AI helped build the tool," not "AI runs at export time."
