1
Agent Submits Invitation Request
Who: Agent or Team Owner (Admin App)
localhost:3101
1
Open the Admin App
Go to http://localhost:3101 in your browser. Use a separate browser window from the CS Console and User App.
2
Log in as Agent or Team Owner
Go to /login. Enter credentials for an account that has OWNER, ADMIN, MANAGER, or SALES role in a workspace. You must be an active org member to submit invitation requests.
Redirected to the workspace dashboard after login.
3
Navigate to Invitations
Click Invitations in the sidebar, or go to /invitations directly. You will land on the "+ New Request" tab by default.
4
Enter the invitee's email address
Type the test user's email address in the Email address field. Use a real email inbox you can access to receive the Gmail onboarding email.
Do not use an email already registered in the system — it will still work, but no onboarding email will be sent (user_existed = true).
5
Select a role
For an AGENT workspace: choose from Sales, Finance, or Individual Subscriber.
For a TEAM workspace: choose from Manager, Admin, or Employee.
Select one by clicking the radio card.
6
Add optional notes (skip if not needed)
The Notes field is visible to CS Console staff when they review the request. Add context like "New sales staff onboarding" if relevant.
7
Read the PDPA data notice
A blue notice appears above the Submit button confirming that by submitting, you acknowledge having the right to share this person's email with myexpensio.
Confirm the blue PDPA notice is visible before submitting. This is the sender disclosure required under Malaysian PDPA 2010.
8
Click "Submit Request"
The button is active only when email + role are filled in.
A green or blue banner appears at the top of the page.
If auto_approve = OFF: "Your request has been sent to the platform team for review." Tab switches to Active. Status = PENDING.
If auto_approve = ON: "Onboarding email sent to [email] with login instructions and a temporary password." Status = EXECUTED → skip to Phase 3.
9
Confirm in the Active tab
Click the Active tab. The request should appear with status Pending. Note the email and role match what you entered.
2
CS Console Reviews & Executes
Who: Platform / Console Staff (CS App)
localhost:3102
1
Open the CS Console in a new browser window
Go to http://localhost:3102. Use a different browser or an incognito window so you don't conflict with the Admin App session.
2
Log in as Console staff
Go to /login. Use credentials for an account with Console (CS) staff access.
Redirected to the CS Console dashboard.
3
Go to Invitation Queue
Click Invitation Queue in the sidebar or navigate to /invitation-queue. The Pending tab is shown by default.
4
Find the request from Phase 1
Locate the row with the email address you just submitted. Check the Workspace, Type, Role, and Requested By columns match what you expect.
Status shows Pending with the correct email and role.
5
Click "Approve & Execute"
Click the green Approve & Execute button on the right side of the row. A confirmation modal will appear.
6
Review the modal and confirm
The modal shows a summary: Email, Workspace, Type, Role. Read the three bullet points — it will create the user, assign them to the workspace, and send an invite email. Click Approve & Execute inside the modal.
7
Wait for execution to complete
The button will show "Executing…" for a moment while it creates the user in Supabase Auth, sets up the profile, assigns workspace membership, and sends the Gmail email.
Green success banner: "User account created — invite email sent successfully."
The request moves to the Executed tab with status Executed.
If the banner says "email_warning" or email_sent = false, the Gmail SMTP credentials may be missing or incorrect. Check GMAIL_SMTP_USER and GMAIL_SMTP_APP_PASSWORD env vars.
8
Check the Audit Log (optional)
Go to Audit in the CS Console sidebar. Find the INVITATION_REQUEST_EXECUTED entry. The metadata should show email_sent: true, user_existed: false.
3
Check the Gmail Onboarding Email
Who: The invited user (check their inbox)
Gmail inbox
1
Open the invitee's Gmail inbox
Check the inbox for the email address you invited. Allow up to 1–2 minutes for delivery. Also check the Spam / Promotions folder if it doesn't appear in Primary.
2
Verify the sender and subject
From: effort.myexpensio@gmail.com
Subject: Your myexpensio account for [Org Name]
Email arrives from the correct Gmail sender with the correct subject.
3
Verify the email body contains all required fields
Check the email contains all of the following:

✓ Greeting with email address or display name
✓ Organisation name
✓ Login URL: http://localhost:3100/login
✓ Email: the invitee's email address
✓ Temporary password: a 10-character code (no ambiguous chars like 0/O/l/I)
✓ Default rate template line
✓ Post-login setup instructions (4 items)
All fields present and readable. Copy the temporary password — you need it in Phase 4.
4
Verify the PDPA legal footer
Scroll to the bottom of the email. Below the signature, after a horizontal line, confirm the following are present:

✓ Sentence explaining their email was added by their organisation admin
✓ Mention of Malaysian Personal Data Protection Act 2010 (PDPA)
✓ Link to Privacy Policy
✓ Link to Terms of Service
✓ Contact email for deletion requests: support@myexpensio.com
This footer is legally required under PDPA before the user has given consent. Confirm it is present on every invite email.
4
First Login with Temporary Password
Who: The invited user (User App)
localhost:3100
1
Open the User App in a fresh browser window
Open a new incognito / private window to avoid any session conflicts with the Admin App or CS Console. Go to http://localhost:3100/login.
2
Enter email and temporary password
Email: the invited email address
Password: the 10-character temporary password from the email (copy-paste to avoid typos)

Click Login or press Enter.
3
Confirm automatic redirect to /change-password
The middleware detects must_change_password: true in the user's auth metadata and immediately redirects to /change-password — you do not reach the dashboard.
Page shows: "Set your new password — This is your first login with a temporary password. Please set your own password before continuing."
If you land on /home instead of /change-password, the must_change_password flag was not set correctly during execution. Check the Supabase auth.users app_metadata for this user.
5
Set a New Password
Who: The invited user (User App)
localhost:3100/change-password
1
Enter a new password
Type a new password in the New password field. Must be at least 8 characters. Use the 👁 toggle to verify the input. Do not reuse the temporary password.
2
Confirm the password
Re-enter the same password in Confirm new password. Both must match exactly.
3
Review and accept the PDPA Data & Privacy Notice
A blue Data & Privacy Notice box appears above the submit button. It explains that myexpensio collects name, email, and expense data solely to provide the service, under the Malaysian PDPA 2010.

✓ Check "I have read and agree to the Terms of Service and Privacy Policy" — this is required and unlocks the submit button.
✓ Optionally check "I agree to receive product updates" for marketing emails.
The "Save password and continue" button is disabled and greyed out until the required checkbox is ticked. Once ticked, the button becomes active.
If the button is clickable without ticking the checkbox, the consent guard has not been applied. Verify change-password/page.tsx state and the complete-first-login API validation.
4
Click "Save password and continue"
The app calls Supabase to update the password, then calls /api/auth/complete-first-login with consent_terms: true and consent_marketing. The API clears the must_change_password flag, writes consent to the profiles table, and writes a PDPA_CONSENT_RECORDED entry to audit_logs.
Green banner: "Password updated successfully. You will be logged out now."
After ~1 second the app signs you out and redirects to /login?passwordChanged=1.
5
Check the login page confirmation
The login page at /login?passwordChanged=1 should show a success notice indicating the password was changed and the user can now log in.
6
Final Login → Dashboard
Who: The invited user (User App)
localhost:3100
1
Log in with the new password
Email: the invited email
Password: the new password you just set (not the temp password)

Click Login.
2
Confirm landing on /home (not /change-password)
The middleware now sees must_change_password = false and routes the user directly to /home. The dashboard loads with the user's workspace data.
User lands on /home and can see the myexpensio dashboard. The workspace name matches the organisation they were invited to.
If redirected back to /change-password, the complete-first-login API did not clear the flag. Check /api/auth/complete-first-login route and Supabase auth.users app_metadata.
3
Verify workspace membership is active
The user should see their workspace in the dashboard. Confirm the role shown (Employee, Sales, etc.) matches what was set in Phase 1. Try navigating to Trips, Claims, or Settings to verify access.
Navigation works and the correct role is displayed. No "access denied" or empty state errors.
4
Check Settings → confirm profile is set up
Go to /settings. The email should be pre-filled. The user can complete the 4 setup items mentioned in the onboarding email: change password (already done), review rate, set department/location, and set company name.

Common Issues & Fixes
If something goes wrong, check here first
SymptomLikely causeFix
Submit Request button stays disabled Email or role not filled in Fill both email and select a role radio card
Request stays PENDING, no auto-execute auto_approve_invitations = false in platform_config Expected — proceed to Phase 2 (CS Console)
CS Console execute fails with "CONFLICT" Request status must be APPROVED before execute Click "Approve" first if the execute button shows separately
Email not received SMTP env vars missing or incorrect; check spam folder Verify GMAIL_SMTP_USER and GMAIL_SMTP_APP_PASSWORD in apps/cs/.env.local
Login fails with temp password Password copied with extra space; user already existed Copy-paste directly; check audit log for user_existed: true
After login, goes to /home not /change-password must_change_password not set in app_metadata Check Supabase auth.users → app_metadata for this user
After changing password, goes to /change-password again complete-first-login API failed to clear the flag Check server logs for the /api/auth/complete-first-login route
User lands on /home but sees no workspace data org_members row missing or status ≠ ACTIVE Check org_members table in Supabase for the user_id
PDPA footer missing from email Old build / cached version of send-onboarding-email.ts Restart the CS app server — the footer was added to the source file
