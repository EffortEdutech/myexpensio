MyExpensio — Subscription Reference

Last updated: 2026-05-14 · v4.0.0 Two independent billing systems · Complete upgrade flow · "Double subscription" explained

1 · Two Completely Independent Systems

🔵 System A — Individual Plan
profiles.subscription_plan
Belongs to the person, not a company.
Controls Personal Space and Business Space access.
Follows the user wherever they go — across workspaces.

Values: FREE  STANDARD ⚠ reserved  PREMIUM

Charged to: the individual (RM 29/mo)
🟢 System B — Organisation Plan
subscription_status.tier
Belongs to the company / workspace, not a person.
Controls Work Space mileage claim quota — shared by all org members.
Completely invisible to solo users with no workspace.

Values: FREE  PRO

Charged to: the organisation (separate Stripe product)
Can a user be PREMIUM and the org be PRO at the same time?
Yes — and it is completely normal. The two systems are in different database tables with different Stripe products and different webhook signing secrets. They never conflict. A business owner who subscribes to Premium personally (RM 29/mo) while their company also pays for a PRO org plan is simply using both. Each payment goes to a separate Stripe subscription and each webhook updates a different table.

Important rule (implemented in code): A user with subscription_plan = PREMIUM automatically gets unlimited Work Space claims — even if their org is on the FREE tier. This means a solo gig worker only needs to pay for PREMIUM once (RM 29/mo), not PREMIUM + org PRO. They are not double-charged. See Section 1b below.

Code: isUnlimited = tier === 'PRO' || subscription_plan === 'PREMIUM'
The rule that prevents double-charging: In api/usage/current/route.ts and api/billing/summary/route.ts, the entitlement check is:
isUnlimited = subscription_status.tier === 'PRO' || profiles.subscription_plan === 'PREMIUM'

Either condition alone is sufficient for unlimited Work Space claims. A solo PREMIUM user does not need to also pay for an org PRO subscription. The org PRO subscription only matters when you need the whole team / all agency members to get unlimited claims.

2 · How Each System Gets Set

Individual Plan — subscription_plan
Sign-up: always starts as FREE (default in profiles table)
→ PREMIUM: user pays via /upgrade page → Stripe Checkout → webhook fires → activatePremiumIndividual() → sets PREMIUM + creates BUSINESS space
Manual override: CS Console → Users → Edit → Individual Subscription Plan (SUPER_ADMIN only) — for support escalations
NOT set during invitation — invitation only adds to org_members, plan stays FREE
Org Plan — subscription_status.tier
Org provisioning: always starts as FREE
→ PRO: org owner pays via Stripe → webhook fires → upsertSubscription() → sets PRO for the whole org
Manual override: CS Console → Workspaces → Edit → Plan tier (SUPER_ADMIN only)
NOT set during invitation — invitation only creates org_members row

3 · How a User Upgrades to Premium — Step by Step

This is the only self-service upgrade path. The user does everything from within the User App. No admin action is needed for a successful upgrade.
Step 1
User hits the gate
User taps Business Space or the upgrade banner anywhere in the app.
PremiumGate component checks subscription_plan — if FREE, shows upgrade modal.
Step 2
Visits /upgrade page
Free vs Premium plan comparison. Taps "Upgrade — RM 15/month" button.
Step 3
Stripe Checkout
POST /api/billing/premium/checkout
Creates Stripe Checkout Session with metadata.plan_type = "PREMIUM_INDIVIDUAL" and metadata.user_id.
Redirects to Stripe-hosted payment page.
Step 4
Stripe fires webhook
Event: checkout.session.completed
Hits: POST /api/billing/webhook
Verified with STRIPE_PREMIUM_WEBHOOK_SECRET
Step 5
activatePremiumIndividual()
1. Sets profiles.subscription_plan = 'PREMIUM'
2. Upserts BUSINESS space for this user
3. Writes audit log entry
Step 6
User lands at /upgrade/success
Business Space is now unlocked. User can navigate to /business immediately.
Important: The webhook uses STRIPE_PREMIUM_WEBHOOK_SECRET — a separate env var from STRIPE_WEBHOOK_SECRET (which is for the org PRO plan). These must both be set in Vercel env vars. Mixing them up will cause signature verification failures and the upgrade will appear to succeed in Stripe but never activate in the database.

Answer to "double subscription" concern: The PREMIUM row and the PRO row in the table above are two completely separate charges. A user being PREMIUM does not interact with or cancel the org's PRO status. A user's org being PRO does not give them Business Space — only their personal subscription_plan = PREMIUM does that. You can think of it like a phone plan vs a gym membership — having both is normal, and cancelling one has no effect on the other.

5 · Stripe Configuration Required

Premium Individual
Product: MyExpensio Premium
Price: RM 15/month recurring
Env var: STRIPE_PREMIUM_MONTHLY_PRICE_ID
Webhook: checkout.session.completed
Secret: STRIPE_PREMIUM_WEBHOOK_SECRET
Webhook URL: myexpensio-jade.vercel.app/api/billing/webhook
Org PRO Plan
Product: MyExpensio PRO
Price: Per-org pricing
Env var: STRIPE_ORG_PRICE_ID (or equivalent)
Webhook events: subscription.updated / deleted / invoice.paid / invoice.payment_failed
Secret: STRIPE_WEBHOOK_SECRET
Webhook URL: Admin app endpoint
⚠ Do Not Confuse
STRIPE_WEBHOOK_SECRET = org PRO webhook
STRIPE_PREMIUM_WEBHOOK_SECRET = individual Premium webhook

These are two separate webhook endpoints registered in Stripe Dashboard with different signing secrets. Swapping them causes silent failures — Stripe reports success but the database never updates.

6 · Manual Override (Support Escalations)

Override individual subscription_plan
Where: CS Console (myexpensio-cs.vercel.app) → Users → click Edit on any user → Individual Subscription Plan section

Who can: SUPER_ADMIN only

When to use: Refund processed but want to keep Premium for 30 days; user reports payment succeeded but plan not activated; fraud / abuse revocation.

Audit trail: Writes USER_SUBSCRIPTION_PLAN_CHANGED event with actor email, old→new plan, and the support note you enter.
Override org subscription_status.tier
Where: CS Console → Workspaces → click Edit on any workspace → Subscription section

Who can: SUPER_ADMIN only

When to use: Trial period extension; payment failed but grace period needed; internal test org needs PRO features.

Audit trail: Writes SUBSCRIPTION_CHANGED event with provider: 'MANUAL' so it's distinguishable from Stripe-driven changes.
